Users pick a plan: personal or business. Business accounts get extra fields for company name and size. When they switch back to personal, those stale values need to be flagged for cleanup. Four rules handle the whole flow.

## The Rules

```ts
import { enabledWhen, requires, umpire } from '@umpire/core'

const signupUmp = umpire({
  fields: {
    email:           { required: true, isEmpty: (v) => !v },
    password:        { required: true, isEmpty: (v) => !v },
    confirmPassword: { required: true, isEmpty: (v) => !v },
    referralCode:    {},
    companyName:     { isEmpty: (v) => !v },
    companySize:     { isEmpty: (v) => !v },
  },
  rules: [
    // Can't confirm what doesn't exist yet
    requires('confirmPassword', 'password'),

    // Company fields only on business plan
    enabledWhen('companyName',
      (_v, cond) => cond.plan === 'business',
      { reason: 'business plan required' }),
    enabledWhen('companySize',
      (_v, cond) => cond.plan === 'business',
      { reason: 'business plan required' }),

    // Company size needs a company name first
    requires('companySize', 'companyName'),
  ],
})
```

Six fields, four rules. The rules read top-to-bottom as English: "confirmPassword requires password. Company fields are enabled when the plan is business. Company size requires company name."

## Step 1 — Personal Plan, Password Entered

```ts
const result = signupUmp.check(
  { email: 'alex@example.com', password: 'hunter2' },
  { plan: 'personal' },
)
```

| Field | enabled | required | reason |
| --- | --- | --- | --- |
| email | `true` | `true` | `null` |
| password | `true` | `true` | `null` |
| confirmPassword | `true` | `true` | `null` |
| referralCode | `true` | `false` | `null` |
| companyName | `false` | `false` | `'business plan required'` |
| companySize | `false` | `false` | `'business plan required'` |

Company fields are off the field on a personal plan. `confirmPassword` is available because `password` is present. Notice that `required` is suppressed to `false` on disabled fields. You cannot require something that is not available.

## Step 2 — Switch To Business Plan

Same values, different conditions:

```ts
const result = signupUmp.check(
  { email: 'alex@example.com', password: 'hunter2' },
  { plan: 'business' },
)
```

Now `companyName` is enabled, but `companySize` is still off:

```ts
result.companyName
// { enabled: true, required: false, reason: null, reasons: [] }

result.companySize
// { enabled: false, required: false,
//   reason: 'requires companyName', reasons: ['requires companyName'] }
```

Two rules layer here. `enabledWhen` opens the gate for the business plan, then `requires` keeps `companySize` waiting until `companyName` has a value. The rules compose cleanly without imperative branching.

## Step 3 — No Password Yet

```ts
const result = signupUmp.check(
  { email: 'alex@example.com' },
  { plan: 'personal' },
)

result.confirmPassword
// { enabled: false, required: false,
//   reason: 'requires password', reasons: ['requires password'] }
```

`password` is absent, so `confirmPassword` is disabled. The `required: false` suppression matters here because a validation layer should not complain about a field that is not even available.

## Step 4 — Switching Plans, Flagging Resets

The user filled the business-only fields, then switched back to personal. The values are still there, but the fields are no longer available. `flag()` recommends the cleanup:

```ts
const penalties = signupUmp.flag(
  { values: { email: 'alex@example.com', password: 'hunter2',
              companyName: 'Acme', companySize: '50' },
    conditions: { plan: 'business' } },
  { values: { email: 'alex@example.com', password: 'hunter2',
              companyName: 'Acme', companySize: '50' },
    conditions: { plan: 'personal' } },
)

// [
//   { field: 'companyName', reason: 'business plan required',
//     suggestedValue: undefined },
//   { field: 'companySize', reason: 'business plan required',
//     suggestedValue: undefined },
// ]
```

Only conditions changed. The values are identical. Because snapshots include `conditions`, `flag()` handles conditions-only transitions naturally.

Three conditions must be true for each recommendation:

1. The field was enabled in the before snapshot but disabled in the after snapshot.
2. The field's current value is not empty, according to its `isEmpty`.
3. The current value differs from `suggestedValue`, which is the field `default` or `undefined`.

Apply the recommendations to converge:

```ts
signupUmp.flag(
  { values: { email: 'alex@example.com', password: 'hunter2',
              companyName: 'Acme', companySize: '50' },
    conditions: { plan: 'personal' } },
  { values: { email: 'alex@example.com', password: 'hunter2' },
    conditions: { plan: 'personal' } },
)
// -> [] (nothing left to clean up)
```

## What This Doesn't Cover

Umpire does not validate that the email is well-formed, the password meets policy, or `confirmPassword` matches `password`. Those are correctness concerns. Umpire handles availability: whether fields should be on the field at all.
