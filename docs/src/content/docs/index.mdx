## Install

```bash
npm install @umpire/core
```

Use `@umpire/core` when you just need pure availability logic. Add `@umpire/react`, `@umpire/signals`, or `@umpire/zustand` when you want an adapter on top.

## What It Does

Umpire answers one question: given the current field values and conditions, which fields should be enabled, required, or flagged for cleanup?

It does not manage state, render UI, validate data correctness, or mutate values for you. It recommends resets with `flag()`, but the consumer decides whether to apply them.

## Signup Example

<div class="umpire-columns">
<div>

```ts title="Define rules"
import { enabledWhen, requires, umpire } from '@umpire/core'

const ump = umpire({
  fields: {
    email:           { required: true, isEmpty: (v) => !v },
    password:        { required: true, isEmpty: (v) => !v },
    confirmPassword: { required: true, isEmpty: (v) => !v },
    referralCode:    {},
    companyName:     {},
    companySize:     {},
  },
  rules: [
    requires('confirmPassword', 'password'),

    enabledWhen('companyName',
      (_v, cond) => cond.plan === 'business',
      { reason: 'business plan required' }),

    enabledWhen('companySize',
      (_v, cond) => cond.plan === 'business',
      { reason: 'business plan required' }),

    requires('companySize', 'companyName'),
  ],
})
```

</div>
<div>

```ts title="Check availability"
// Personal plan — company fields disabled
ump.check(
  { email: 'alex@example.com', password: 'hunter2' },
  { plan: 'personal' },
).companyName
// → { enabled: false, reason: 'business plan required' }

// Business plan — companySize requires companyName
ump.check(
  { email: 'alex@example.com', password: 'hunter2' },
  { plan: 'business' },
).companySize
// → { enabled: false, reason: 'requires companyName' }

// Switch plans — what should reset?
ump.flag(
  { values: { companyName: 'Acme', companySize: '50' },
    conditions: { plan: 'business' } },
  { values: { companyName: 'Acme', companySize: '50' },
    conditions: { plan: 'personal' } },
)
// → [
//   { field: 'companyName', suggestedValue: undefined },
//   { field: 'companySize', suggestedValue: undefined },
// ]
```

</div>
</div>

This example is intentionally about availability, not validation. `confirmPassword` depends on `password` being present, but Umpire does not check whether the two values match. That belongs in your validation layer.

Three rule types in action: **`enabledWhen`** (injuries scratch players from the roster), **`oneOf`** (platoon matchups swap batters when the opposing pitcher changes), and **`requires`** (Morrison needs rest days before he can start). ## How the Demo Works

The lineup card above is a React component backed by a real `@umpire/core` instance. Here's how the state flows:

```ts title="Rules — declared once"
const lineupUmp = umpire({
  fields,
  rules: [
    // Platoon: lefty/righty splits at 1B and LF
    oneOf('firstBasePlatoon', {
      vsRighty: ['delgado'],
      vsLefty:  ['vega'],
    }, {
      activeBranch: (_v, cond) => cond.opposingPitcher === 'L' ? 'vsLefty' : 'vsRighty',
    }),

    // Morrison can't pitch without rest
    requires('morrison', 'morrisonRested'),

    // Injuries scratch players
    ...playerIds.map(id =>
      enabledWhen(id, (values) => !values[`${id}_injured`])
    ),
  ],
})
```

```ts title="Check — called on every state change"
const availability = lineupUmp.check(values, { opposingPitcher })
// availability.delgado.enabled → false when opposing pitcher is LHP
// availability.morrison.reason → 'requires morrisonRested'
```

```ts title="Flag — compare two snapshots to find stale values"
// After any state change, compare the previous snapshot to the current one.
// flag() finds fields that just became disabled but still hold non-empty values.
const prevSnapshot = { values: prevValues, conditions: prevConditions }
const currentSnapshot = { values, conditions }

const penalties = lineupUmp.flag(prevSnapshot, currentSnapshot)
// User flips the opposing pitcher from RHP → LHP:
// → [{ field: 'delgado', reason: 'platoon matchup', suggestedValue: undefined }]
// Delgado was in the lineup, but the platoon rule just disabled him.
// The penalty recommends clearing his slot — the UI shows the yellow flag.
```

`check()` tells you what's available *right now*. `flag()` tells you what *just broke* — fields that were enabled in the previous state but disabled in the current state, and still hold values that should be cleaned up. You call both on every state change: `check()` to derive the UI, `flag()` to surface reset recommendations.

## What It Does Not Do

- It does not own form state.
- It does not validate that values are correct.
- It does not hide your fields or render your UI.
- It does not clear stale values automatically.
